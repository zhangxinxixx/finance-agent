# Obsidian Rules

仅在需要写入 Obsidian、更新路线图/任务看板/版本记录/ADR/开发日志，或处理知识库归档时读取。

## 基本规则

- Obsidian vault 固定为：`/home/zxx/wiki/Finance-Agent-Knowledge-Vault`。
- repo 内 `hermes/` 只保留当前可执行状态和临时计划；长期规划、架构决策、版本记录、复盘沉淀一律写入 vault。
- Obsidian 是完整文档权威源；Mem0 只存摘要和约束。
- 不要重新创建旧英文目录，如 `02-Projects`、`03-Architecture`、`04-Data-Sources`、`06-Agent-Workflows`、`_system`、`_templates`。
- 不要为同一类状态新增第二入口；能追加或覆盖现有入口就不要新建平行页面。

## 唯一跟踪入口

| 信息类型 | 唯一维护页 | 规则 |
|---|---|---|
| 项目首页 / 常用入口 | `02-项目/金融分析系统/00-首页.md` | 只放定位、核心链路、唯一跟踪入口和稳定参考入口 |
| 当前进度 | `02-项目/金融分析系统/02-当前进度.md` | 覆盖更新，只写当前事实、已完成、下一步 |
| 当前任务 | `02-项目/金融分析系统/06-任务看板.md` | 覆盖更新，只写 Now / Done / Later |
| 开发路线图 | `02-项目/金融分析系统/04-开发路线图.md` | 覆盖更新，只写阶段和方向 |
| 版本/变更记录 | `02-项目/金融分析系统/09-版本记录.md` | 完成一个切片后追加，必须带验证结果 |
| 风险与卡点 | `02-项目/金融分析系统/08-风险与卡点.md` | 覆盖更新，只保留仍影响后续执行的问题 |
| 知识库整理规则 | `02-项目/金融分析系统/29-知识库整理与归档清单.md` | 覆盖更新，记录整理口径 |
| 架构事实 | `03-架构/总体架构.md`、`03-架构/前端页面架构.md`、`03-架构/后端服务架构.md`、`03-架构/任务调度架构.md`、`03-架构/Agent架构.md` | 覆盖更新，只写稳定边界和当前接口 |
| 每日过程 | `07-开发日志/YYYY-MM-DD.md` | 新建或追加，只记录当天流水和验收 |

禁止把“下一步 / 当前剩余 / 待执行”继续写入旧计划页、源文档页、模块页或每日日志。旧计划如需保留，只在页首标注 `superseded`、`archived`、`source-only`、`legacy` 或 `partial-current`，并指向唯一入口。

## 写入时机

必须写入 vault：

- 完成一个 Phase 或 Task 组 -> 写版本记录；若有架构决策则写 ADR。
- 修改了主链架构或模块边界 -> 写 ADR。
- 发现并解决重要风险/卡点 -> 更新风险与卡点页。
- 每周或里程碑节点 -> 更新当前进度页。

低风险规则文件自身维护、单文件文案修正、格式调整可不新建执行方案，但若形成长期约定，应更新版本记录或对应规则页。

## 版本记录模板

追加到 `02-项目/金融分析系统/09-版本记录.md`：

```markdown
## YYYY-MM-DD - 变更标题

- 类型：feature / fix / arch / refactor / docs
- 范围：[受影响模块]
- 变更：
  - [具体做了什么]
- 验证：[通过的测试/验证命令]
- 影响：[对主链/API/前端的影响]
- 后续：[遗留问题或下一步]
```

## ADR 模板

新建 `03-架构/ADR/ADR-YYYYMMDD-标题.md`：

```markdown
---
tags: [ADR, 架构决策]
date: YYYY-MM-DD
status: accepted
---

# ADR-YYYYMMDD：标题

## 背景
## 决策
## 理由
## 影响
## 状态
```

## 写入命令

优先用 `obsidian` CLI：

```bash
obsidian append vault="Finance-Agent-Knowledge-Vault" path="02-项目/金融分析系统/09-版本记录.md" content="..."
```

CLI 不可用时可直接操作文件系统。禁止写入 `storage/`、`hermes/staging/`、`模板/`、`系统/` 目录，禁止在 vault 存放 `.py` / `.ts` 代码文件。

## 归档与删除口径

- 旧计划、源文档和历史方案默认软归档，不直接删除。
- `status: archived` 表示不再参与当前决策。
- `status: superseded` 表示已被新文档替代。
- `status: source-only` 表示只能查历史。
- `status: legacy` 表示模板或骨架保留但不主动使用。
- 可以物理删除的仅限空 README、明确占位且无链接引用的页面、重复导入且内容已被新页完全覆盖的原始文件。
- 删除前必须先查反链或文本引用；不确定时先软归档。
